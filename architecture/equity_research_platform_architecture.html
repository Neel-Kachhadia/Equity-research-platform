Section One
System Overview & Design Philosophy

1.1 Problem Framing in Institutional Context

Modern institutional equity research operates under a profound information asymmetry problem. Analysts at tier-one firms process thousands of data points daily — quarterly filings, earnings transcripts, management commentaries, regulatory disclosures, macroeconomic releases, and continuous price signals — yet synthesis is largely manual, slow, and subject to cognitive bias. The average analyst covers 12–20 stocks with tools that are fundamentally fragmented: Bloomberg terminals for market data, internal models in Excel, PDFs opened in Acrobat, and email chains for collaborative annotation.

EREBUS is designed to resolve this fragmentation. It is not a data aggregator, and it is not a dashboard. It is an intelligence substrate — a unified reasoning layer that ingests multi-modal evidence, processes it through a rigorous quantitative and linguistic pipeline, and delivers analyst-grade outputs with full traceability. Every claim is grounded. Every score is explained. Every recommendation carries its reasoning chain.

Core Thesis

The competitive advantage of a tier-one research desk is not access to data — it is the quality, speed, and defensibility of synthesis. EREBUS operationalizes that synthesis into a reproducible, auditable, institutional-grade system.

1.2 Design Philosophy

Principle 01
Grounding Over Generation
Every insight must be traceable to a source — a specific page, paragraph, earnings quarter, or data point. The system refuses to generate claims it cannot ground. Hallucination is treated as a critical failure mode, not an edge case.
Principle 02
Explainability as First-Class Output
Scores, rankings, and signals are not outputs — they are inputs to explanation. For every quantitative result, a human-readable reasoning trace is mandatory. SHAP values, feature contributions, and source evidence accompany every conclusion.
Principle 03
Separation of Fast and Deep
Normal Mode (sub-2-second responses for universe-wide signals) and Deep Research Mode (minutes-long, multi-document synthesis for single companies) are architected as distinct pipelines with different latency SLAs, compute profiles, and confidence thresholds.
Principle 04
Institutional Reliability
The system must behave predictably under adversarial conditions: missing data, contradictory filings, management obfuscation. Confidence intervals, data quality scores, and uncertainty flags are mandatory metadata on all outputs.
Principle 05
Multi-Modal by Default
Financial ratios, price series, document text, table data from filings, sentiment from earnings calls, and regulatory risk signals are fused at the feature layer — not bolted on post-hoc. The quant model and NLP model co-inform each other.
Principle 06
Auditability and Compliance
All outputs are versioned. Every model run is logged with its input data snapshot, model version, and timestamp. This supports regulatory audit trails and enables back-testing of research outputs against subsequent market outcomes.

Product Architecture & User Flows

2.1 User Personas

The Senior Equity Analyst
Covers 15–20 stocks, produces quarterly initiation reports, prepares briefing notes for fund managers, responds to management queries about specific positions. Deep domain expertise; wants the system to accelerate and validate, not replace, their judgment.
Core Jobs-to-Be-Done
  • Rapidly identify which of their covered companies has had a material signal change
  • Synthesize earnings call language against prior-period guidance commitments
  • Generate draft initiation reports from uploaded annual reports + financial data
  • Validate their qualitative thesis against quantitative factor evidence
Primary Failure Modes to Avoid
  • Being given confident outputs without sourcing (unverifiable claims)
  • Having to re-do formatting work the system should handle
  • Missing a key risk flagged in a regulatory filing
  • Factor scores that are unexplained black boxes
The Quantitative Portfolio Manager
Manages a factor-based or multi-strategy portfolio. Needs universe-wide factor scores, cross-sectional comparisons, systematic signal generation, and exportable data for integration into their own portfolio optimization models.
Core Jobs-to-Be-Done
  • Run full NIFTY 50 factor scorecard on demand
  • Export structured factor data to Excel or JSON for portfolio optimizer
  • Monitor regime changes (momentum shift, risk-off signals)
  • Compare peer groups on composite alpha scores
Primary Failure Modes to Avoid
  • Stale data presented as current
  • Factor definitions that are inconsistently applied across companies
  • No API access for model integration
  • Cannot drill into what drives a specific ranking position
The Chief Investment Officer / Research Head
Needs high-level dashboards, exception alerts, and synthesis of team research. Reviews deep research reports before investment committee. Wants confidence that outputs are defensible and audit-ready.
Core Jobs-to-Be-Done
  • Daily market intelligence brief: top movers, risk alerts, signal changes
  • Pre-IC review of analyst reports with evidence grounding visible
  • Cross-portfolio risk concentration analysis
  • Management credibility scoring for key investee companies
Primary Failure Modes to Avoid
  • System produces outputs that can't be defended in IC meeting
  • No visibility into confidence levels or data quality
  • Can't generate a polished PDF report on short notice

2.2 User Flows

Normal Mode Flow — Market Intelligence Snapshot

1
Dashboard Landing — Universe Overview
User arrives at the EREBUS dashboard. The system has pre-computed factor scores for the full NIFTY 50 universe (refreshed on schedule or on-demand). They see: composite score ranking, momentum heatmap, sector distribution, top signals.
2
Filter & Sort — Signal Discovery
User filters by sector, applies factor thresholds (e.g., Quality Score > 70, Momentum Rank top 20%), and sorts by composite alpha signal. The ranked list updates in under 500ms using pre-computed feature store data.
3
Company Quick View — Signal Card
Clicking a company opens a signal card: composite score, factor breakdown (Quality / Value / Momentum / Risk), 12-month return chart, recent news headline sentiment, and quick-access link to Deep Research Mode. All computed from cache — no live query.
4
Export Universe Scorecard
User exports the full ranked universe as Excel (structured factor table with all sub-scores) or JSON (API-ready). Export includes timestamp, data vintage, model version, and confidence flags per company.

Deep Research Mode Flow — Institutional Analysis

1
Research Session Initiation
User selects a company (or enters a custom ticker/name). Optionally uploads documents: annual report PDFs, earnings call transcripts, custom Excel financial model. System detects file types, begins parallel ingestion across all sources.
2
Document Processing Pipeline (Background)
Uploaded PDFs are parsed (layout-aware extraction), tables extracted and structured, text semantically chunked, entities tagged (metrics, risks, forward guidance, management commitments). Chunks embedded and indexed into vector store. Progress bar visible to user.
3
Structured Analysis Sections Generated
System automatically generates: (a) Financial Deep Dive — 5-year ratio analysis with trend decomposition; (b) Management Commentary Analysis — guidance vs actual tracking, sentiment scoring; (c) Risk Diagnostic — regulatory, operational, balance sheet; (d) Peer Benchmarking.
4
Natural Language Query Interface
User can ask questions: "What has management said about capex guidance over the past 3 years vs actual spend?" System retrieves relevant chunks from vector store, fuses with structured financial data, generates a grounded response with source citations (document, page, section).
5
Report Compilation & Export
User clicks "Generate Report." System assembles a PDF research report: executive summary, factor scorecard, financial analysis, qualitative insights, risk diagnostics, management credibility score, charts, and full source citation appendix. Exported as PDF, Excel supplement, or JSON API payload.

Section Four
Mode Architecture — Normal vs Deep
Normal Mode
Market Intelligence Layer
Design Goal

Sub-2-second responses for universe-wide queries. Data is pre-computed and cached. No live model inference during request handling.

Trigger

User views dashboard, requests universe scorecard, queries single-company signal card, exports ranked list.

Data Sources
  • Pre-computed feature store (Redis + TimescaleDB)
  • Latest market price data (T+0)
  • Latest fundamental snapshot (T+1)
  • Pre-aggregated sentiment index
Processing Pipeline
  • Cache hit check (Redis, TTL 5 min)
  • Feature store retrieval (≤ 50ms)
  • Score aggregation and ranking (≤ 200ms)
  • Response serialization
Output
  • Ranked universe table with composite scores
  • Factor breakdowns (Quality, Value, Momentum, Risk)
  • Signal cards (Strength, Risk, Trend)
  • Heatmaps and sparklines
Latency Budget

Cache hit: ≤ 200ms. Cache miss (recompute): ≤ 2s. P99 target: ≤ 3s under peak load.

Deep Research Mode
Institutional Analysis Layer
Design Goal

Comprehensive, multi-source, document-grounded analysis. Accepts cold (no prior data) or warm (existing company data) sessions. Designed for investigative depth, not speed.

Trigger

User initiates research session for a specific company, optionally uploads documents, requests analyst-grade report generation.

Data Sources
  • User-uploaded documents (PDFs, Excel, text)
  • Historical financial database (5–10 years)
  • SEBI filings, EDGAR documents (auto-fetched)
  • Earnings call transcripts (vendor API)
  • News corpus (last 24 months, indexed)
  • Pre-computed feature store (supplemental)
Processing Pipeline
  • File ingestion and type detection
  • PDF parsing → chunk → embed → index
  • Historical financial computation (all ratios)
  • NLP pipeline (sentiment, entity, guidance)
  • Multi-source RAG retrieval and synthesis
  • Reasoning engine fusion and report generation
Output
  • Full analyst research report (PDF, 15–25 pages)
  • Factor scorecard with historical trends
  • Management credibility score + evidence
  • Risk diagnostic with source citations
  • Guidance vs actual deviation analysis
  • Interactive charts (downloadable)
Latency Budget

Warm session (existing index): ≤ 90 seconds. Cold session (upload + index + full analysis): ≤ 4 minutes. Async job with progress streaming via WebSocket.

4.1 Mode Routing Logic

# orchestration/mode_router.py

class ModeRouter:
    """
    Routes incoming research requests to either the Normal Mode pipeline
    (fast, pre-computed) or Deep Research Mode pipeline (multi-modal, async).

    Decision is based on: request_type, document_uploads, time_horizon,
    and user role permissions.
    """

    MODE_THRESHOLDS = {
        "max_companies_normal": 50,  # beyond this, always async
        "deep_trigger_horizon_years": 2,  # historical data > 2Y triggers deep
        "min_doc_size_mb_for_async": 1.0,
    }

    def route(self, request: ResearchRequest) -> ModeDecision:
        # Explicit deep mode request
        if request.mode == "deep":
            return ModeDecision(mode="deep", async_job=True)

        # Document upload always triggers deep mode
        if request.uploaded_documents:
            return ModeDecision(
                mode="deep",
                async_job=True,
                reason="document_upload"
            )

        # Historical horizon check
        if request.time_horizon_years > self.MODE_THRESHOLDS["deep_trigger_horizon_years"]:
            return ModeDecision(mode="deep", async_job=True)

        # Single company, no uploads, current data → Normal Mode
        if len(request.company_ids) == 1 and not request.uploaded_documents:
            return ModeDecision(
                mode="normal",
                async_job=False,
                cache_eligible=True
            )

        # Universe query → Normal Mode with pre-computed scores
        if request.universe_query:
            return ModeDecision(mode="normal", async_job=False)

        return ModeDecision(mode="normal", async_job=False)

6.1 File Type Detection

File type detection uses a three-layer strategy: MIME type from HTTP headers (unreliable, used as first pass), python-magic library for magic byte inspection (reliable), and extension-based fallback. Each detected type routes to a type-specific parser.

# ingestion/file_detector.py
import magic
from enum import Enum
from pathlib import Path

class FileType(Enum):
    PDF     = "application/pdf"
    EXCEL   = "application/vnd.openxmlformats-officedocument.spreadsheetml.sheet"
    CSV     = "text/csv"
    TEXT    = "text/plain"
    WORD    = "application/vnd.openxmlformats-officedocument.wordprocessingml.document"
    UNKNOWN = "unknown"

class FileTypeDetector:
    MAGIC_TO_TYPE = {
        "application/pdf": FileType.PDF,
        "application/vnd.openxmlformats": FileType.EXCEL,  # prefix match
        "text/csv": FileType.CSV,
        "text/plain": FileType.TEXT,
    }

    def detect(self, file_bytes: bytes, filename: str) -> FileType:
        # Primary: magic bytes
        mime = magic.from_buffer(file_bytes[:4096], mime=True)
        for prefix, ftype in self.MAGIC_TO_TYPE.items():
            if mime.startswith(prefix):
                return ftype

        # Fallback: extension
        ext = Path(filename).suffix.lower()
        ext_map = {".pdf": FileType.PDF, ".xlsx": FileType.EXCEL,
                   ".xls": FileType.EXCEL, ".csv": FileType.CSV,
                   ".txt": FileType.TEXT, ".docx": FileType.WORD}
        return ext_map.get(ext, FileType.UNKNOWN)

6.2 Parsing Strategies

PDF Parsing

PDF parsing uses a two-path strategy depending on the PDF type detected (digital native vs scanned):

Path A — Digital Native PDF
  • Primary parser: pdfplumber (layout-aware, preserves table structure)
  • Fallback: PyMuPDF (fitz) for complex layouts
  • Table detection: bounding box heuristics + line detection
  • Table extraction: pdfplumber.extract_tables() → DataFrame
  • Text extraction: paragraph-boundary detection via whitespace analysis
  • Section detection: heading style analysis (font size, bold flags)
Path B — Scanned / Image PDF
  • Detection: entropy analysis + embedded text ratio check
  • OCR Engine: Tesseract 5.x (primary), AWS Textract (high-value documents)
  • Pre-processing: Deskew (OpenCV), denoise, contrast normalization
  • Table OCR: AWS Textract AnalyzeDocument (AnalyzeFeatureType.TABLES)

7.1 Semantic Chunking Strategy

Naive fixed-size chunking destroys semantic coherence in financial documents (e.g., splitting a table mid-row or cutting a paragraph mid-argument). EREBUS uses a hierarchical semantic chunking strategy:

1
Document Segmentation
Identify major structural sections using heading detection (font metadata for digital, layout heuristics for OCR output). Create top-level segments: "MD&A", "Financial Statements", "Risk Factors", "Directors' Report", etc.
2
Paragraph-Boundary Chunking
Within each section, split at paragraph boundaries. Target chunk size: 400–600 tokens (empirically optimal for financial text retrieval). Apply sliding window overlap of 50 tokens to preserve cross-paragraph context.
3
Table Handling — Separate Track
Tables are extracted separately and NOT split. Each table becomes one chunk with structured metadata: column headers, row labels, cell values as key-value pairs (serialized). Tables also generate a natural-language summary chunk for semantic search.
4
Chunk Enrichment
Each chunk receives: section_title, page_number, chunk_index, parent_doc_id, contains_guidance flag (pattern-matched), contains_table flag, entity_metrics list (extracted numbers + metric names), sentiment_score (FinBERT inference).

7.3 Management Credibility Engine

A critical differentiator of EREBUS is its ability to track management guidance against actual outcomes over time. This creates a Management Credibility Score — a quantified measure of how accurately management has historically predicted their own company's performance.

MCScompany = 100 × [1 − mean(|GuidanceDeviationt|/|Guidancet|)] × CredibilityWeight

where:
GuidanceDeviationt = Actualt − Guidancet
CredibilityWeight = f(consistency, direction_accuracy, time_horizon)

Guidance statements are extracted from earnings call transcripts and annual report MD&A sections, tagged with the metric, period, and value. The system then cross-references against actual reported financials in the database, computing deviation statistics per guidance commitment.

8.2 Feature Normalization Pipeline

# features/normalization.py
import numpy as np
from scipy.stats import rankdata

class CrossSectionalNormalizer:
    """
    Normalizes features cross-sectionally within each universe refresh.
    Uses rank-based normalization to be robust against outliers —
    critical in financial data where extreme values are common.
    """

    def normalize(self, feature_matrix: pd.DataFrame, universe: str) -> pd.DataFrame:
        # Step 1: Winsorize at 2.5%/97.5% to handle outliers
        winsorized = feature_matrix.apply(
            lambda col: col.clip(
                lower=col.quantile(0.025),
                upper=col.quantile(0.975)
            )
        )

        # Step 2: Cross-sectional rank transformation → [0, 100]
        # Rank percentile is more robust than z-score for non-normal distributions
        ranked = winsorized.apply(
            lambda col: (rankdata(col, method='average') - 1) / (len(col) - 1) * 100
        )

        # Step 3: Handle missing values via sector-median imputation
        # (never impute with 50th percentile blindly — use sector peers)
        for col in ranked.columns:
            if ranked[col].isna().any():
                sector_medians = ranked.groupby('sector')[col].transform('median')
                ranked[col] = ranked[col].fillna(sector_medians)

        # Step 4: Invert risk features (higher raw risk = lower score)
        INVERT_FEATURES = ['beta_1y', 'volatility_60d', 'debt_to_equity', 'accruals_ratio']
        for feat in INVERT_FEATURES:
            if feat in ranked.columns:
                ranked[feat] = 100 - ranked[feat]

        return ranked

9.1 Multi-Factor Composite Alpha Score

The composite alpha signal is a weighted combination of four factor scores, each of which is a cross-sectionally ranked composite of sub-factors. Weights are calibrated using information coefficient (IC) analysis and updated quarterly. A Barra-style factor model structure is used for orthogonality.

Composite Alpha Score
αcomposite = wQ·SQ + wV·SV + wM·SM − wR·SR + wNLP·SNLP

where all Si ∈ [0, 100] (cross-sectional percentile rank)

Default weights: wQ = 0.35, wV = 0.25, wM = 0.20, wR = 0.15, wNLP = 0.05
Subject to: Σwi = 1.0; weights reviewed and recalibrated quarterly via IC-weighted optimization
Quality Factor Score (S_Q)
SQ = 0.30·Rank(ROE) + 0.25·Rank(ROIC) + 0.20·Rank(FCF_Conv) + 0.15·Rank(GM_Stability) + 0.10·Rank(100 − Accruals)

All Rank() operations are cross-sectional within the trading universe on the computation date.
Value Factor Score (S_V)
SV = 0.40·Rank(100 − EV/EBITDA_rank) + 0.30·Rank(EarningsYield) + 0.20·Rank(P/B_adj) + 0.10·Rank(DividendYield)

Note: EV/EBITDA is inverted (lower multiple = better value rank).
Momentum Factor Score (S_M)
SM = 0.40·Rank(Ret12-1M) + 0.30·Rank(EPS_RevMomentum) + 0.20·Rank(RevSurprise) + 0.10·Rank(RelStrength)

Momentum is computed over the t-12M to t-1M window (Jegadeesh-Titman formulation with 1-month skip to avoid short-term reversal).
Risk Factor Score (S_R) [Penalizer — Higher = More Risk]
SR = 0.30·Rank(Beta) + 0.30·Rank(IdioVol) + 0.25·Rank(D/E_trend) + 0.15·Rank(100 − InterestCoverage)

SR is used as a PENALTY in the composite score. High risk score reduces alpha_composite.

9.2 Factor IC Calibration

# models/factor_calibration.py
import numpy as np
from scipy.optimize import minimize

class FactorWeightOptimizer:
    """
    Quarterly recalibration of factor weights using Information Coefficient (IC)
    from historical backtests. Maximizes IC-weighted signal while maintaining
    diversification across factors.
    """

    def compute_ic(self, factor_scores: pd.DataFrame, 
                        forward_returns: pd.Series, horizon_months: int = 3) -> dict:
        # Rank IC (Spearman correlation of factor score rank vs forward return rank)
        ic_results = {}
        for factor in ['quality', 'value', 'momentum', 'risk', 'nlp']:
            ic, p_value = spearmanr(
                factor_scores[factor].rank(),
                forward_returns.rank()
            )
            ic_results[factor] = {
                'ic': ic,
                'ic_ir': ic / factor_scores[factor].std(),  # IC Information Ratio
                't_stat': ic * np.sqrt(len(factor_scores)) / np.sqrt(1 - ic**2),
                'significant': p_value < 0.05
            }
        return ic_results

    def optimize_weights(self, ic_results: dict) -> dict:
        # Maximize IC-weighted composite; constrain weights ≥ 0.05 for diversification
        ic_vec = np.array([ic_results[f]['ic'] for f in ['quality', 'value', 'momentum', 'risk', 'nlp']])
        ic_ir_vec = np.array([ic_results[f]['ic_ir'] for f in ['quality', 'value', 'momentum', 'risk', 'nlp']])

        result = minimize(
            lambda w: -np.dot(w, ic_ir_vec),  # maximize IC IR
            x0=np.ones(5) / 5,
            constraints=[{'type': 'eq', 'fun': lambda w: np.sum(w) - 1.0}],
            bounds=[(0.05, 0.50)] * 5  # 5% min, 50% max per factor
        )
        return dict(zip(['quality', 'value', 'momentum', 'risk', 'nlp'], result.x))

            WHERE dr.company_id = %s
            ORDER BY ge.guidance_for_period

            top_negative_drivers=[...],   # top 3 factors dragging the score
        )

    def explain_factor_score(self, company_id: str, factor: str) -> FactorExplanation:
        # Rule-based attribution for transparent factor scores
        # Attribution = weight_i × (sub_score_i − universe_median_i)
        sub_scores = self.get_sub_scores(company_id, factor)
        universe_medians = self.get_universe_medians(factor)
        weights = FACTOR_WEIGHTS[factor]

        attributions = {
            sub: weights[sub] * (sub_scores[sub] - universe_medians[sub])
            for sub in sub_scores
        }

        return FactorExplanation(
            factor=factor,
            composite_score=sum(weights[s] * sub_scores[s] for s in sub_scores),
            attributions=attributions,
            interpretation=self._generate_interpretation(attributions, factor)
        )

# orchestration/deep_research_orchestrator.py
import asyncio

async def run_deep_research(session: ResearchSession) -> ResearchResult:
    """
    Parallelizes independent analysis steps to minimize total wall-clock time.
    Document indexing, financial computation, and news retrieval run concurrently.
    """
    # Step 1: File ingestion must complete first (sequential)
    ingestion_results = await ingest_uploaded_files(session.uploaded_files)

    # Step 2: Run independent pipelines in parallel
    quant_task = asyncio.create_task(compute_financial_analysis(session.company_id))
    index_task = asyncio.create_task(index_documents(ingestion_results))
    news_task  = asyncio.create_task(retrieve_news_intelligence(session.company_id))
    peer_task  = asyncio.create_task(compute_peer_benchmarking(session.company_id))

    # Broadcast progress via WebSocket as tasks complete
    quant_result, index_result, news_result, peer_result = await asyncio.gather(
        quant_task, index_task, news_task, peer_task,
        return_exceptions=True  # don't fail all if one step fails
    )

    # Step 3: Reasoning engine (sequential — requires all step 2 outputs)
    reasoning_result = await run_reasoning_engine(
        quant=quant_result,
        documents=index_result,
        news=news_result,
        peers=peer_result,
        session=session
    )

    # Step 4: Report generation (sequential — requires reasoning output)
    report = await generate_report(reasoning_result, session.report_config)

    return ResearchResult(session_id=session.session_id, report=report)

REFUSAL_THRESHOLDS = {
    "min_data_coverage": 0.60,   # Refuse ranking if < 60% features available
    "min_rag_relevance": 0.65,   # Refuse RAG answer if best chunk score < 0.65
    "min_faithfulness": 0.70,    # Quarantine if NLI faithfulness < 0.70
    "max_data_staleness_days": 45  # Warn if fundamental data > 45 days old
}

class RefusalEngine:
    def evaluate_rag_response(self, response: LLMResponse, 
                                  retrieved_chunks: list[Chunk]) -> RefusalDecision:
        # Check 1: Retrieval relevance threshold
        max_relevance = max(chunk.relevance_score for chunk in retrieved_chunks)
        if max_relevance < REFUSAL_THRESHOLDS["min_rag_relevance"]:
            return RefusalDecision(
                should_refuse=True,
                reason="INSUFFICIENT_DATA",
                message=f"No document chunk with sufficient relevance found (max score: {max_relevance:.2f}). "
                        "To answer this question, please upload: [specific document type]."
            )

        # Check 2: Faithfulness score (NLI cross-encoder)
        faithfulness = self.faithfulness_scorer.score(response.text, retrieved_chunks)
        if faithfulness < REFUSAL_THRESHOLDS["min_faithfulness"]:
            return RefusalDecision(
                should_refuse=True,
                reason="LOW_FAITHFULNESS",
                message="Response faithfulness below threshold. Quarantined for review."
            )

        # Check 3: Numerical cross-verification
        numbers_in_response = extract_numbers(response.text)
        for num, context in numbers_in_response:
            if not self._verify_number_in_context(num, retrieved_chunks):
                response.add_warning(f"Unverified number: {num} in context: '{context}'")

        return RefusalDecision(should_refuse=False)
